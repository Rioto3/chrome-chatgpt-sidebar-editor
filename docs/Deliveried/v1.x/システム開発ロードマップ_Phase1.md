# システム開発ロードマップ_Phase1

## **はじめに**

この文書は、別文書「現状システム仕様書」と同時点から、
現在の最終目標であるサービス外販までについて整理しています。

**基本方針として「段階的開発」を採用**しています。

各フェーズで確実に成果を検証しながら、システム導入の成功率を高め、
ビジネスを安全かつコスト効率よく展開することを可能にします。

以下３点のテーマで順に説明いたします：

1. 必要な機能
2. コスト・スケジュール感
3. 問題提起

## **1. 必要な機能**

### **1. 基本機能**
企業等の問い合わせフォームからの応募活動をサポートするシステムです。

#### **【システムの価値】**
- 作業範囲の定量化：
スプレッドシートで作業範囲を定義することで作業が定量化されるため、
作業の適切な評価・作業のアウトソーシング を期待できます。

- 反復作業コストの削減（ユーザのフォーム遷移・フォーム入力を補助）: 
作業時間効率の向上・操作間違いを防ぎ作業モチベーション低下を防止 を期待できます。

#### **【技術上必要な要素】**
- スプレッドシートと連携してデータを読み込む機能
- 企業等のURLに遷移する機能
- 問い合わせフォームに遷移する機能
- 問い合わせフォームに半自動入力する機能
- 半自動入力情報を入力し安全に保持する機能
- 視覚的に理解しやすいUI

### **2. トラッキングシステム**
応募側が用意するURLの使用状況をマネジメントするシステムです。

#### **【システムの価値】**
- 応募活動のフィードバック：
クリック回数・期間などのデータを表示確認できることで、活動のフィードバックを取得できます。
フィードバックは戦略設計や他意思決定のために活用可能です。

#### **【技術上必要な要素】**
- トラッキング結果を表示できる機能:（例: /insightページ・スプレッドシート）
- トラッキングURLを作成できる機能: サーバ機能
- トラッキングURLを運用する機能: サーバ機能

### **3. 販売システム** 
外部顧客に対してサービスを提供・課金するシステムです。

#### **【システムの価値】**
- サービス価値の公開・価値共有：
ツールを一般公開することによって、営業業務の効率化という観点から経済に貢献します。

- サービス価値の市場価値：
サービスの価値に対する適切な対価を得ることで、継続的な開発・サポート体制・高品質なサービスを維持し、市場のフィードバックを受けます。

#### **【技術上必要な要素】**
- 認証機能(必須)：限定されたユーザのみ許可する機能
- 課金・決済連携機能：ユーザに権限を販売する機能
- ユーザー管理機能：ユーザの権限の許可・限定許可・無効化・期限管理などを行う機能

## **2. コスト・スケジュール感**

### **大まかな工数見積もり**
- **基本システムの完全完成**: ２週間 / ３万円
- **トラッキングシステム完成**: ３週間 / ５万円
- **外販開始**: ８週間 / ※ トラッキングシステム完成後見積り

※ 上記の分類にこだわらず**必要機能追加**の段階的な実現を推奨します。

### **コスト・工数について**
最新状況・これまでの開発展開から見積もっております。
開発中の軽微な要件追加・変更については承ります。

外販に関わる業務上の基本的なサポート体制については、
お客様の方で対応可能となるよう設計いたします。
（顧客管理・支払管理など）

### **著作権について**
安全な活動と責任の観点からお客様には、
ソフトウェアの販売権全権の譲渡をお約束いたします。

ソフトウェアの著作権については、
開発活動と責任の観点から「Rioto3」が保持いたします。

一般公開時「© Rioto3」の表記をお願いいたします。

## **3. 問題提起**

### **ビジネス課題**
- 価格設定・課金モデルの検討（月額/年額/従量制/買切）
- 緊急対応（サービス利用不可時など）
- 外販ユーザーサポート体制の構築（問い合わせ対応・トラブルシューティング）
- プロダクトキー管理体制の構築（管理者・管理方法）
- プロダクトキー販売体制の構築（販売方法・売上管理方法）
- ドメインネームの更新管理
- ソフトウェアライセンス

### **技術的課題**
- バックエンドサーバの長期保守・運用体制の確立
- スケーラビリティ対応（ユーザー数急増時の処理能力）
- セキュリティ要件の継続的対応・脆弱性管理
- エラーハンドリング・障害対応の自動化
- データベースのバックアップ・復旧体制
- ドメインネーム契約切れエラー対応
- バックエンドサーバダウン時のエラー対応
- Google認証サーバダウン時のエラー対応

### **設計的課題**
- トラッキングの可視化方法の確立（ウェブアプリ開発・スプレッドシート等）
- ユーザー体験の最適化（設定画面・操作フローの改善・主婦などでも使いやすい）
- 本文テンプレート機能の実装（効率化・品質向上）

### **設計的アイデア**
- データ分析項目の拡張（アクセス地域・アクセス時間・使用端末などの測定指標）
- データ分析機能の追加（ランキング作成・統計値作成等）
- マルチブラウザ対応の検討（Chrome以外への展開）
- レポート機能の充実（成果の可視化・分析）
- カスタマイズ機能の追加（業界特化・企業特化）

### **選定技術**
- DOMパターンによるフォーム検索・入力ロジック：高速・無コスト・安定（AI実装と比較）
- Cloudflare Worker：エッジコンピューティングによる高速バックエンド処理・サービス安定性・無料枠
- Cloudflare D1データベース：Workerとの相性・高速キャッシング
- Hono：安全API設計
- js+Webpack：高速開発
- github：バージョン管理（Private設定）

## **参考：【段階的開発】について**

業務やビジネスでシステムを扱う上で、
ユーザの要望は頻繁に変更されることが自然だと考えています。

技術サイドで最大限お応えするために、
以下の観点でプロジェクトの提案・ご案内をしております。

### **背景**
- システム開発側は、タイムリーに要望に応えるべき
- 長い開発期間は、完成時のシステム要件とユーザ需要との乖離を生む

### **段階的開発の採用**
各段階でゴールのみに集中し、その瞬間ビジネスに必要なものを届けます。

### **開発原則**
- **最小機能での価値提供** → 検証 → 次段階
- **要件発散の防止** → 明確なゴール設定
- **継続的改善** → 実運用フィードバック重視

### **Lean RAD-TDD-RDDプロセスモデル**
段階的開発については、一般規格手法を参考にしています：

- **Lean Startup**: 市場投入・フィードバック・改善方式で良いものをつくります
- **Rapid Application Development**: ユーザに近いものを優先的に完成を目指します
- **Test Driven Development**: 早期開発でも高品質を保証します。
- **Recovery-Driven Design**: 高リスク範囲については慎重に対応します
